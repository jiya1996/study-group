# OpenClaw 定时采集任务优化复盘

> 2026-03-28 | 从方案 A 到方案 B 的完整迭代记录

---

## 背景

利用腾讯云硅谷服务器上部署的 OpenClaw（OpenClaw Bot），自动化采集 AI + 金融领域资讯，服务于自媒体运营和 BudVest 产品。

---

## 方案 A（v1）：按主题拆分 + 12h 固定窗口

### 配置
- 早报 08:00：AI 官方发布 + arXiv + GitHub（搜索过去 12h）
- 午报 13:00：KOL 推文 + HN + Newsletter（搜索过去 12h）
- 晚报 22:00：美股 + 融资 + 宏观政策（搜索过去 12h）

### 6 个问题
1. **按主题拆分** → 同一事件跨多个日报，用户需翻看 3 条
2. **12h 窗口重叠** → 早报和午报有 7h 重叠，信息重复
3. **晚报 22:00 太晚** → 用户 21:00 已休息
4. **窗口无设计依据** → 没有考虑硅谷时区的产出高峰
5. **信源不全** → 缺少金融和中文 AI 信源
6. **信息量不足** → 每次只有 8-12 条混合输出

---

## 方案 A（v2）：统一采集 + 滚动窗口

### 改进
- 统一 6 板块采集，不按主题拆分
- 滚动时间窗口：11h + 5h + 8h = 24h，无重叠
- 晚报从 22:00 调整到 21:00
- 每板块最多 20 条，信息量 10x 提升
- 新增金融信源 + 中文 AI 信源

### 实测结果
- MiniMax-M2.5 联网搜索能力弱
- 只能搜到 GitHub Trending，其他板块全部缺失
- 会把训练数据伪装成实时搜索（旧新闻当新推送）
- Twitter/X 需登录无法访问
- 硅谷服务器访问国内网站受限

### 结论
**Prompt 设计没问题，LLM 能力不足。** 方案 A 保留配置但禁用，等 OpenClaw 支持更强的 LLM 后可重新启用。

## 方案 B：Claude Schedule（当前方案）

### 架构
- Claude Sonnet 4.6 远程 Agent（Anthropic Cloud）
- WebSearch 工具实时搜索全网
- 结果保存为 Markdown，commit 到 GitHub
- 每天 3 次：08:00 / 13:00 / 21:00

### 优势
- 真正的联网搜索能力
- 能访问 Twitter/X、中文信源、Newsletter
- 不会用训练数据伪装实时结果
- 搜索结果有真实链接

### 待验证
- 搜索覆盖面是否达到 6 板块要求
- 输出质量和格式稳定性
- 长期运行的可靠性

## 设计原则总结

### 核心思想
信息采集系统的设计应该"从消费端反推"：
1. 先确定"谁在什么时候看"（用户作息）
2. 再确定"什么时候信息最多"（硅谷高峰）
3. 最后确定采集窗口

### 三个原则
1. **统一采集不分主题** — 用户只关心"过去几小时发生了什么重要的事"
2. **时间窗口无缝衔接** — 24 小时全覆盖，不重叠不遗漏
3. **板块化输出** — 采集统一但输出分类，方便快速跳转

### 教训
- **先验证 LLM 能力再设计方案** — 方案 A 的 Prompt 设计花了很多时间，但 LLM 根本执行不了
- **MVP 先跑起来** — 不要一开始就搞复杂架构（采集脚本 + 中间存储），先用最简单的方式验证
- **A/B 测试** — 新旧方案并存，用数据说话
